# cc-connect Session Management Investigation

This document explains the technical logic behind session management in **cc-connect**, specifically how `/list`, `/switch`, and `/delete` commands are implemented and how they interact with the AI agent backends (like Gemini).

## 1. Overview of Session Layers

`cc-connect` maintains two distinct session concepts:
1.  **Platform Session (SessionKey):** A unique identifier for a user on a specific platform (e.g., `telegram:{chatID}:{userID}`).
2.  **Agent Session (AgentSessionID):** A unique identifier generated by the AI agent CLI (e.g., Gemini's `sessionId`).

The `Engine` maps a **Platform Session** to an **Agent Session** via a local JSON session store.

---

## 2. Listing Sessions (`/list`)

**Logic Location:** `Engine.cmdList` in `core/engine.go`

1.  **Fetch from Agent:** The engine calls `agent.ListSessions()`.
    - For **Gemini**, this involves scanning `~/.gemini/tmp/<project_slug>/chats/*.json` files.
    - It filters out internal "subagent" sessions and those without user messages.
2.  **Enrich with Metadata:** The engine combines the agent's session data (ID, summary, message count) with its own local session names (set via `/name`).
3.  **Display Rendering:**
    - **Plain Text:** Returns a numbered list with markers (▶ for active, ◻ for inactive).
    - **Interactive Cards:** (On supported platforms like Feishu) Renders a scrollable card with "Switch" buttons.

---

## 3. Switching Sessions (`/switch <id|name|index>`)

**Logic Location:** `Engine.cmdSwitch` and `Engine.matchSession` in `core/engine.go`

1.  **Resolution (Matching):** The engine tries to find a match for the user's query in the following priority:
    - **Numeric Index:** Matches the number shown in the most recent `/list` output.
    - **Custom Name:** Matches a name set by the user via `/name`.
    - **ID Prefix:** Matches the start of the hex Agent Session ID.
2.  **State Cleanup:**
    - Calls `Engine.cleanupInteractiveState()`.
    - Terminates the current running agent process (if any) for that user.
3.  **Mapping Update:**
    - Updates the active `AgentSessionID` for that `SessionKey` in the local store.
    - Clears the engine's local turn history (since we are starting fresh with a resumed context).
4.  **Resumption:** The next user message will trigger a new CLI process with the `--resume <new_id>` flag.

---

## 4. Deleting Sessions (`/delete <id|index|batch>`)

**Logic Location:** `Engine.cmdDelete` in `core/engine.go`

1.  **Capability Check:** Checks if the current agent implements the `SessionDeleter` interface.
2.  **Target Selection:**
    - Supports **Single Delete:** `/delete 1` or `/delete abc123`.
    - Supports **Batch Delete:** `/delete 1,3,5` or `/delete 1-5`.
    - Supports **Interactive Mode:** If the platform supports cards, it opens a "Delete Mode" UI with multi-select checkboxes.
3.  **Backend Execution:** Calls `agent.DeleteSession(sessionID)`.
    - For **Gemini**, the agent implementation searches for the corresponding `.json` file in the Gemini chats directory and deletes it from disk.
4.  **Local Sync:** If the deleted session was the *active* one, the engine resets the mapping for that user, effectively starting a "new" session on the next turn.

---

## 5. Persistence

- **Local Metadata:** `cc-connect` saves mappings and custom names in `data/sessions/<project_name>_<hash>.json`.
- **Agent Data:** The actual conversation history remains managed by the agent CLI (e.g., in `~/.gemini/`). `cc-connect` only stores the "pointer" (the ID).
